# Environment & Data Promotion

How code, schema, and data move between local, staging, and production.

---

## Operating Model

### Three things move independently

| What | How it moves | Tool |
|---|---|---|
| **Code** | Git branches: develop → staging → main | `git merge`, PR |
| **Schema** | Alembic migrations: same migration files run per-env | `alembic upgrade head` |
| **Reference data** | Curated snapshots: export → import | `scripts/promote_reference_data.py` |

These are **never mixed**. A code deploy does not bring data. A data promotion does not change code.

### What is reference data (promotable)

Data that the app needs to produce valuations but is NOT generated by user activity.

| Table | Source | Promotable? |
|---|---|---|
| `product` | Crawler, pipeline identification | **Yes** |
| `market_comparable` | Crawler, pipeline market search | **Yes** |
| `new_price_snapshot` | Crawler, pipeline new-price lookup | **Yes** |

### What is environment-local (never promoted)

Data generated by users or by the environment's own runtime.

| Table | Source | Promoted? |
|---|---|---|
| `valuations` | User valuations | **No** — per-environment |
| `price_snapshots` | Pipeline snapshots | **No** |
| `product_embedding` | SigLIP embeddings | **No** |
| `agent_job` | Agent crawl jobs | **No** |
| `price_observation` | Agent observations | **No** |
| `training_sample` | VALOR ETL | **No** |
| `valor_model` | VALOR training | **No** |
| `valor_estimate` | VALOR predictions | **No** |
| `price_statistic` | VALOR stats | **No** |

### Why this separation matters

- Staging gets clean, curated data to test against — not a messy copy of dev
- Production can start empty and get the same curated snapshot
- User feedback in staging doesn't contaminate production, and vice versa
- Each environment owns its own VALOR training cycle

## Git Workflow

```
develop    →    staging    →    main
(daily)         (test)          (production)
```

| Branch | Deploys to | Auto-deploy? | Who merges |
|---|---|---|---|
| `develop` | (local only) | No | Developer |
| `staging` | Railway staging | Yes | Developer merges develop → staging |
| `main` | Railway production | Yes | Owner merges staging → main via PR |

### Daily flow

1. Work on `develop`, commit, push
2. When ready to test on staging: `git checkout staging && git merge develop && git push`
3. Railway auto-deploys staging
4. Run migrations: `railway run -e staging bash -c "cd backend && alembic upgrade head"`
5. Promote reference data:
   `STAGING_DATABASE_URL="..." python scripts/promote_reference_data.py import --target staging`
6. Verify staging:
   `STAGING_DATABASE_URL="..." python scripts/promote_reference_data.py verify --target staging`
7. Test staging
8. When staging is stable: open PR staging → main, merge
9. Railway auto-deploys production
10. Run migrations: `railway run -e production bash -c "cd backend && alembic upgrade head"`
11. Promote: `PRODUCTION_DATABASE_URL="..." python scripts/promote_reference_data.py import --target production`

## Data Promotion Workflow

### Step 1: Export from local

```bash
# Crawl locally first (if needed)
python scripts/crawl_prices.py --priority 1 --max 20

# Export curated snapshot
python scripts/promote_reference_data.py export
```

This creates `data/snapshots/YYYYMMDD_HHMMSS/` with:
- `products.jsonl`
- `comparables.jsonl`
- `new_prices.jsonl`
- `manifest.json` (counts, filters, product keys, timestamp)

A `latest` symlink always points to the most recent snapshot.

### Step 2: Import into staging

```bash
STAGING_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py import --target staging
```

This is **idempotent**:
- Products: UPSERT on `product_key` (updates brand/model/last_seen)
- Comparables: UPSERT on `listing_url` (updates price/active/last_seen)
- New prices: INSERT only if no duplicate (product_key + source + fetched_at)

Running it twice is safe. Running it with updated data merges cleanly.

### Step 3: Verify

```bash
STAGING_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py verify --target staging
```

Checks: migration status, pgvector, row counts, data quality, env-local tables empty.

### Step 4: Later — same for production

```bash
PRODUCTION_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py import --target production
```

Same snapshot, same script, different target.

## Safety Guards

### Strict environment variables (no fallback)

| Target | Required env var | Falls back to DATABASE_URL? |
|---|---|---|
| staging | `STAGING_DATABASE_URL` | **No — hard error** |
| production | `PRODUCTION_DATABASE_URL` | **No — hard error** |
| export (local) | `DATABASE_URL` | Yes — defaults to localhost |

The script also rejects URLs containing `localhost` or `127.0.0.1` for remote targets.

### What is NOT promoted

`valuation_count` is excluded from product exports. It's a runtime counter incremented per valuation — environment-specific, not reference data. Target environments start at 0 and count their own.

---

## Quality Filters

The export applies these filters (configurable via flags):

| Filter | Default | Flag |
|---|---|---|
| Product seen within N days | 30 | `--max-age-days` |
| Min unflagged comparables per product | 3 | `--min-comparables` |
| Comparable last_seen within N days | 60 | `--comparable-max-age` |
| Comparable price range | 200–150,000 SEK | Hardcoded |
| Comparable not flagged | Always | — |
| Comparable is active | Always | — |
| New price snapshot age | 14 days | Hardcoded |

---

## Idempotency Guarantees

| Table | Key | Behavior on re-import |
|---|---|---|
| `product` | `product_key` (PK) | Updates brand, model, category, last_seen |
| `market_comparable` | `listing_url` (UNIQUE) | Updates price, is_active, last_seen |
| `new_price_snapshot` | product_key + source + fetched_at | Skips exact duplicates |

No IDs from local are carried over. Staging/production auto-increment their own IDs.

## How DATABASE_URL Works Across Environments

Railway injects `postgres://user:pass@host:port/db`. Three different parts of the codebase consume it:

| Consumer | Needs | Conversion |
|---|---|---|
| FastAPI app runtime | `postgresql+asyncpg://` | `config.py:_normalize_database_url()` handles automatically |
| Alembic migrations | `postgresql+psycopg2://` | `alembic/env.py` converts automatically |
| Promotion script | `postgresql+psycopg2://` | `promote_reference_data.py:to_sync_url()` converts automatically |

You never need to manually rewrite URLs. Set `DATABASE_URL` (or the target-specific var) and each tool normalizes it.

### pgvector extension

`CREATE EXTENSION IF NOT EXISTS vector` is run in two places for safety:
1. **Migration `ebfc16bbbee6`** runs it as the first step — this is the primary path.
2. **`database.py:init_db()`** also runs it on app startup as a belt-and-suspenders check.

Railway's default PostgreSQL template may or may not include pgvector. Verify after provisioning (see RAILWAY_STAGING_SETUP.md Step 2). If not available, use Docker image `pgvector/pgvector:pg16`.

## What NOT To Do

- Do NOT `pg_dump` the full local database and `pg_restore` into staging
- Do NOT TRUNCATE all tables before import (destroys staging-generated data)
- Do NOT copy valuations, feedback, or embeddings between environments
- Do NOT use `DATABASE_URL` for staging/production imports — use the target-specific variable
- Do NOT run `alembic upgrade head` without verifying which environment you're targeting

## Weekly Promotion Runbook

### Phase 1: Prepare locally

```bash
# 1. Crawl fresh market data
python scripts/crawl_prices.py --priority 1 --max 20

# 2. Export curated snapshot
python scripts/promote_reference_data.py export

# 3. Review what will be promoted
cat data/snapshots/latest/manifest.json
```

### Phase 2: Deploy and promote to staging

```bash
# 4. Merge code
git checkout staging && git merge develop && git push

# 5. Wait for Railway auto-deploy (or: railway up -e staging)

# 6. Run migrations
railway run -e staging bash -c "cd backend && alembic upgrade head"

# 7. Promote reference data
STAGING_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py import --target staging

# 8. Verify
STAGING_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py verify --target staging

# 9. Smoke test
curl https://staging-url.up.railway.app/health
curl -X POST https://staging-url.up.railway.app/value \
  -H "Content-Type: application/json" \
  -d '{"brand":"Sony","model":"WH-1000XM4"}'

# 10. Create Railway backup (Dashboard → Postgres → Backups)
```

### Phase 3: Promote to production (when staging is stable)

```bash
# 11. Merge code
git checkout main && git merge staging && git push
# (or: PR staging → main, merge in GitHub)

# 12. Wait for Railway auto-deploy

# 13. Run migrations
railway run -e production bash -c "cd backend && alembic upgrade head"

# 14. Promote the SAME approved snapshot
PRODUCTION_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py import --target production

# 15. Verify
PRODUCTION_DATABASE_URL="postgresql://..." \
  python scripts/promote_reference_data.py verify --target production

# 16. Create Railway backup
```

The same snapshot file is used for both staging and production. Staging is the proving ground. If staging is good, production gets the identical data.
